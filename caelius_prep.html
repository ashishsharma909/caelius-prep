01 — Core Technical
Java & Object-Oriented Programming
The 4 Pillars of OOP
Must Know▼
OOP is a programming style where you model real-world things as objects. An object has attributes (what it has) and methods (what it can do). Example: a Car object has color and speed (attributes), and can accelerate and brake (methods).
1. Encapsulation — Hiding internal data
Wrapping data (variables) and methods inside a class, and restricting direct access to the data from outside. You expose only what is necessary through public methods (getters/setters). Think of a bank account — you cannot directly change the balance; you must go through the deposit() or withdraw() methods.
class BankAccount { private double balance; // hidden from outside public double getBalance() { return balance; // controlled access } public void deposit(double amount) { if (amount > 0) balance += amount; // validation inside } }
Interview answer: "Encapsulation protects data integrity. By making variables private and providing controlled access through methods, we prevent invalid states — for example, you can't set a negative balance directly."
2. Inheritance — Reusing existing code
A child class inherits attributes and methods from a parent class using the extends keyword. This avoids code duplication and models "is-a" relationships. A Dog IS-A Animal, so Dog can inherit all Animal behaviors and add its own.
class Animal { String name; void eat() { System.out.println(name + " is eating"); } } class Dog extends Animal { void bark() { System.out.println("Woof!"); } } // Dog inherits eat() automatically Dog d = new Dog(); d.name = "Bruno"; d.eat(); // inherited from Animal d.bark(); // Dog's own method
3. Polymorphism — Same interface, different behavior
Poly = many, morphism = forms. One method name behaves differently depending on the context. Two types:
Method Overloading (Compile-time): Same method name, different parameters in the same class.
Method Overriding (Runtime): Child class provides its own version of a parent class method.
// Overloading — same class, different params class Calculator { int add(int a, int b) { return a + b; } double add(double a, double b) { return a + b; } } // Overriding — child redefines parent's method class Animal { void sound() { System.out.println("..."); } } class Cat extends Animal { @Override void sound() { System.out.println("Meow"); } }
4. Abstraction — Hiding complexity
Showing only the essential features and hiding implementation details. When you drive a car, you use the steering wheel and pedals — you don't need to know how the engine works. Achieved through Abstract classes and Interfaces.
abstract class Shape { abstract double area(); // no implementation here void display() { System.out.println("Area: " + area()); } } class Circle extends Shape { double radius; Circle(double r) { this.radius = r; } double area() { return 3.14 * radius * radius; } }

Exception Handling
High Priority▼
An exception is an unexpected event that disrupts program flow (file not found, null pointer, divide by zero). Java handles them with try-catch-finally.
try { int result = 10 / 0; // throws ArithmeticException } catch (ArithmeticException e) { System.out.println("Error: " + e.getMessage()); } finally { System.out.println("This always runs"); // cleanup code }
Checked exceptions: Must be handled at compile time (IOException, SQLException). You MUST either catch them or declare them with throws.
Unchecked exceptions: Runtime exceptions — NullPointerException, ArrayIndexOutOfBoundsException. Not mandatory to catch but should be.
02 — Enterprise Technologies
REST APIs & HTTP
What is REST and how does it work?
Must Know▼
REST (Representational State Transfer) is an architectural style for building web services. It defines rules for how clients and servers communicate over HTTP. Think of it as the "language" that web applications use to talk to each other.
The 6 REST Constraints
1. Stateless: Each request must contain all the information needed. The server does not store session state between requests. Every call is independent.
2. Client-Server: Frontend (client) and backend (server) are separate. They communicate only through APIs. This separation allows them to evolve independently.
3. Uniform Interface: Consistent URL patterns and HTTP methods. Resources are identified by URLs.
4. Cacheable: Responses can be cached to improve performance.
5. Layered System: Client doesn't need to know if it's talking directly to the server or through a load balancer, cache, etc.
6. Code on Demand (Optional): Server can send executable code (like JavaScript) to the client.
HTTP Methods — The Core Four
GET /users → Fetch all users (read only, no body) GET /users/42 → Fetch user with id=42 POST /users → Create a new user (send data in body) PUT /users/42 → Replace user 42 completely (full update) PATCH /users/42 → Update only some fields of user 42 DELETE /users/42 → Delete user 42

JSON vs XML — Data Exchange Formats
High Priority▼
Both are formats for sending structured data between systems. JSON is modern and lightweight; XML is older but still widely used in enterprise.
JSON — JavaScript Object Notation
{ "employee": { "name": "Ashish Sharma", "age": 21, "skills": ["Java", "REST", "Python"], "active": true, "address": { "city": "Bhiwani", "state": "Haryana" } } }
Rules: Keys must be in double quotes. Values can be string, number, boolean, null, array, or object. Lightweight, easy to parse, natively supported in JavaScript.

04 — Architecture
SOA & Message Queues
Service-Oriented Architecture (SOA)
High Priority▼
SOA is a software design approach where an application is built as a collection of independent services. Each service performs one specific business function and communicates with others through well-defined APIs.
Real-world example — E-commerce order
When you place an order on Amazon: Order Service → validates and stores the order Payment Service → charges your credit card Inventory Service → checks and reserves stock Shipping Service → arranges delivery Notification Service → sends you a confirmation email Each service is independent. Payment Service doesn't care HOW Notification Service works — it just calls its API.
Your TrustLens project was SOA: Audio Processing Service (handled file input), ML Inference Service (ran the model), Results Service (stored and returned results), and the Next.js frontend consumed all of these via REST APIs. This is exactly what you should say in the interview.
SOA vs Microservices
SOA: Services are larger, may share a common data bus or ESB (Enterprise Service Bus). More structured, common in large enterprises.
Microservices: Extremely fine-grained services, independently deployable, communicate over lightweight protocols. Modern approach.
Think of SOA as "departments in a company" and microservices as "individual employees, each with one clear role".
Message Queues (MQ)
High Priority▼
A message queue enables asynchronous communication between services. The sender (producer) puts a message in the queue and moves on — it does not wait. The receiver (consumer) picks up the message when ready and processes it.
Synchronous vs Asynchronous Communication
Synchronous (HTTP/REST): Client → "Process this order" → Server Client ← [waits 3 seconds] ← Server Client ← "Done! Order #1234" ← Server (Client is blocked while waiting) Asynchronous (Message Queue): Producer → drops message in queue → Queue Producer continues doing other work... Consumer ← picks up message later ← Queue Consumer → processes and sends result separately
Why use Message Queues?
Decoupling: Services don't need to know about each other — they just read from the queue.
Reliability: If a service is down, messages wait in the queue safely. Nothing is lost.
Load handling: If 10,000 orders arrive at once, the queue absorbs the load. Processing happens at a manageable pace.
Popular tools: RabbitMQ, Apache Kafka (high throughput), IBM MQ (enterprise, used with Salesforce/MuleSoft)
Interview answer: "Message queues provide asynchronous, loosely-coupled communication between services. For example, in an order processing system, when a customer places an order, the Order Service drops a message into the queue and immediately confirms to the customer. The Inventory and Shipping services pick up and process these messages independently. This makes the system more resilient — if the Shipping Service goes down, orders keep queueing safely."

Bonus Points▼

Anypoint Platform: MuleSoft's main tool — where you design, build, test, and deploy integrations (called Mule applications).
Mule Flow: A sequence of processing steps. Data enters, gets transformed, and exits. Like an assembly line for data.
Connectors: Pre-built plugins to connect to popular systems — Salesforce Connector, Database Connector, HTTP Connector, Email Connector. You drag and drop, no need to write connection code.
DataWeave: MuleSoft's transformation language. Converts data from one format to another — JSON to XML, rename fields, filter data.
API-led Connectivity: MuleSoft's recommended architecture with 3 layers — System APIs (raw data from backend), Process APIs (business logic), Experience APIs (tailored for mobile, web, partners).

Java & OOP — Technical Panel Questions
Q1 What are the four pillars of OOP? Give a real-world example for each. Critical
Encapsulation: Bundling data and methods, restricting direct access. Example: A bank account where balance is private — you can only modify it through deposit() and withdraw() methods.

Inheritance: Child class inherits from parent. Example: Dog extends Animal — Dog gets eat() and breathe() for free, plus adds its own bark().

Polymorphism: Same interface, different behavior. Example: Animal.sound() returns "..." but Cat.sound() returns "Meow" and Dog.sound() returns "Woof" — same method call, different output.

Abstraction: Hiding implementation, showing only what's needed. Example: You use Arrays.sort() without knowing whether it uses TimSort or QuickSort internally.
In my TrustLens project, I used encapsulation in the AudioProcessor class — the raw model weights were private, and I exposed only a public analyze(audioData) method. This is a real production example of encapsulation protecting the internal ML logic.
Q2 What is the difference between an interface and an abstract class? When would you use each? Critical
An abstract class can have concrete methods, instance variables, and constructors. A class can extend only one abstract class. Use it when you have a parent-child relationship with shared code.

An interface defines only a contract — what methods must exist, not how they work (in Java 8+, default methods are allowed). A class can implement multiple interfaces. Use it when unrelated classes need to share a capability.

I would use an abstract class for Vehicle → Car, Truck because they share common fuel() and maintenance() logic. I would use an interface for Serializable or Flyable — both an Airplane and a Bird can fly, but they have nothing else in common, so it makes no sense for them to share a parent class.
Q3 Explain method overloading vs method overriding with an example. High
Overloading is compile-time polymorphism — same method name in the same class, different parameter lists. The compiler decides which version to call based on the arguments you pass.

Overriding is runtime polymorphism — a subclass provides its own implementation of a method already defined in the parent class. The JVM decides which version to call at runtime based on the actual object type.
Overloading: print(String s) and print(int n) in the same class — same name, different input. Overriding: Animal has sound() returning "...", but Dog overrides it with @Override sound() returning "Woof" — the parent method is replaced by the child's version.
Q4 What is the difference between ArrayList and LinkedList? High
ArrayList is backed by a dynamic array. Random access by index is O(1) — very fast. But inserting or deleting in the middle is O(n) because elements need to shift.

LinkedList stores elements as nodes where each node points to the next. Inserting at the beginning or end is O(1). But accessing element by index requires traversal — O(n).

Rule of thumb: Use ArrayList when you need frequent random access. Use LinkedList when you need frequent insertions and deletions, especially when implementing a queue or stack.
Q5 What is the difference between checked and unchecked exceptions in Java? High
Checked exceptions are checked at compile time. The compiler forces you to either catch them or declare them with throws. Examples: IOException, SQLException, FileNotFoundException. They represent conditions outside the program's control.

Unchecked exceptions are subclasses of RuntimeException. The compiler doesn't force you to handle them. Examples: NullPointerException, ArrayIndexOutOfBoundsException, IllegalArgumentException. They usually represent programming errors.

When reading a file, I use checked IOException because the file might not exist — this is an external condition. NullPointerException is unchecked because it means I made a programming mistake by not checking for null before calling a method.
Q6 What is the difference between HashMap and Hashtable? High
HashMap: Not thread-safe (not synchronized), allows one null key and multiple null values, faster. Modern choice.

Hashtable: Thread-safe (synchronized), does not allow null keys or values, slower due to synchronization overhead, legacy class.

If you need thread safety, use ConcurrentHashMap instead of Hashtable — it's synchronized at the bucket level rather than the entire map, making it much more performant in concurrent environments.
REST APIs & HTTP Questions
Q7 What is REST? What are its key principles? Critical
REST (Representational State Transfer) is an architectural style for designing networked applications. It relies on stateless, client-server communication using the HTTP protocol.

Key principles: Stateless (each request contains all needed information), Client-Server separation (frontend and backend evolve independently), Uniform Interface (consistent URL and HTTP method conventions), Cacheable (responses can be cached), Layered System (client doesn't know about intermediaries), Code on Demand (optional — server can send executable code).
In my TrustLens project, I built a REST API where the frontend and ML backend were completely decoupled. The frontend sent POST /analyze with audio data and received a JSON response — it had no idea whether the model was Python-based or what algorithm was used. That's REST's client-server separation in action.
Q8 What is the difference between PUT and PATCH? High
PUT replaces the entire resource. If you send PUT /users/1 with only the name field, all other fields (email, phone) get wiped out unless you include them.

PATCH partially updates a resource. If you send PATCH /users/1 with only the name field, only the name changes — email and phone stay intact.

Rule: Use PUT for full replacement, PATCH for partial updates. In practice, most APIs use PATCH for update operations because it is safer.
Q9 What does stateless mean in REST? Why is it important? High
Stateless means the server does not store any client session information between requests. Every request must contain all the information needed to process it — authentication token, parameters, and body data.

Why important: Scalability — any server instance can handle any request since there is no session state to worry about. Reliability — if one server goes down, requests go to another without data loss. Simplicity — easier to debug since each request is self-contained.

This is why REST APIs use tokens (JWT) in each request header rather than server-side sessions.
Q10 What HTTP status code would you return when a user successfully creates a new resource? High
201 Created — not 200 OK. The distinction matters: 200 means the request succeeded and here is the data. 201 specifically means a new resource was created as a result of the request. The response should include either the Location header pointing to the new resource, or the newly created resource in the response body.
SOA, MQ & Architecture Questions
Q11 What is Service-Oriented Architecture? How is it different from microservices? High
SOA is a design approach where an application is built as a collection of loosely coupled, independent services that communicate through APIs. Each service performs a specific business function.

Microservices is an evolution of SOA with even finer-grained services, independently deployable, each with its own database, typically communicating over lightweight HTTP or message queues.

Key difference: SOA services tend to be larger and may share an Enterprise Service Bus. Microservices are smaller, more independent, and communicate directly. SOA is the concept; microservices is a more modern, granular implementation.
My TrustLens project followed SOA principles — the audio service, ML service, and results service were independent and communicated through REST APIs. If I had split these into even smaller services (separate services for audio validation, format conversion, feature extraction), that would have been microservices.
Q12 When would you use a message queue instead of direct REST API calls? High
Use a message queue when:
1. The operation is slow and you don't want the client to wait (async processing)
2. You need to handle traffic spikes without crashing the system (buffering)
3. Services should be loosely coupled — the sender doesn't need to know if the receiver is available
4. You need guaranteed delivery even if a service goes down temporarily

Example: In an order system, placing an order should be instant. The actual fulfillment (inventory check, shipping arrangement) can happen asynchronously via a queue. If the shipping service is temporarily down, orders just queue up — nothing is lost.
Q13 What is JSON? What are its advantages over XML? High
JSON (JavaScript Object Notation) is a lightweight data-interchange format that is easy for humans to read and write, and easy for machines to parse and generate. It uses key-value pairs and arrays.

Advantages over XML: Less verbose (smaller payload size), faster to parse, natively supported in JavaScript without additional libraries, simpler syntax. XML is still preferred in enterprise contexts for its schema validation capabilities (XSD), namespace support, and when working with legacy SOAP services or SAP systems.

Q16 What is SOQL? How is it different from SQL? High
SOQL (Salesforce Object Query Language) is used to query data from Salesforce objects. It is similar to SQL but specifically designed for Salesforce's data model.

Key differences: You query Salesforce Objects instead of database tables. No JOIN keyword — you traverse relationships using dot notation (Contact.Account.Name). No SELECT * — you must specify all fields. SOQL is read-only (no INSERT/UPDATE in SOQL — those use DML statements in Apex).

SELECT Id, Name, Email FROM Contact WHERE Account.Industry = 'Technology' — this uses relationship traversal (Account.Industry) instead of a JOIN, which is the main syntax difference from SQL.

MuleSoft uses an API-led connectivity model with three layers: System APIs (connect to raw data sources), Process APIs (business logic and transformation), Experience APIs (optimized for specific consumers like mobile apps or portals).
Additional Technical Questions
Q18 What is the difference between a process and a thread? High
A process is an independent program execution with its own memory space. A thread is a lightweight unit of execution within a process that shares the process's memory.

Multiple threads in the same process can communicate easily (shared memory) but need synchronization to avoid race conditions. Multiple processes are isolated from each other, making them safer but communication between them is more expensive (inter-process communication).

Java's Thread class and Runnable interface allow multithreading. In modern Java, ExecutorService manages thread pools efficiently.
Q19 What is an API? Why are APIs important in enterprise software? High
An API (Application Programming Interface) is a defined contract that allows two software systems to communicate. It specifies what requests can be made, in what format, and what responses to expect — without exposing internal implementation.

Enterprise importance: Large companies have dozens of systems (CRM, ERP, HR, e-commerce). APIs allow these systems to share data and automate workflows without rebuilding them. APIs also enable third-party integrations — a company can expose its data to partners through APIs. This is why companies like Salesforce and MuleSoft (Caelius's core platforms) are built entirely around the API economy.
Q20 What is the difference between authentication and authorization? High
Authentication verifies identity — "Who are you?" The system confirms you are who you claim to be (via password, token, biometric).

Authorization verifies permissions — "What are you allowed to do?" After confirming your identity, the system checks what resources and actions you have access to.

Example: You log in to Salesforce (authentication). Once logged in, your profile determines whether you can view all records or only your team's records (authorization). HTTP 401 = authentication failure. HTTP 403 = authorization failure.
Q21 What is a design pattern? Name two common ones. High
A design pattern is a reusable solution to a commonly occurring problem in software design. They are templates, not code — you adapt them to your situation.

Singleton: Ensures only one instance of a class exists. Example: database connection pool, logging service. Useful when a single shared resource must be controlled carefully.

Factory: Creates objects without specifying the exact class. A factory method returns an object based on input. Example: ShapeFactory.create("circle") returns a Circle object. Useful when the exact type is determined at runtime.
Q22 Explain what a database index is and why it matters. High
An index is a data structure that speeds up data retrieval operations in a database. Without an index, a database must scan every row to find matching data (full table scan). With an index, it can jump directly to the relevant rows — similar to a book's index allowing you to jump to a topic without reading every page.

Trade-off: Indexes speed up SELECT queries but slow down INSERT, UPDATE, and DELETE operations because the index must also be updated. They also use additional disk space.

Best practice: Index columns that are frequently used in WHERE clauses, JOIN conditions, and ORDER BY clauses. Avoid over-indexing.
Q23 What is Git and what is the difference between git merge and git rebase? High
Git is a distributed version control system. It tracks changes in code, allows multiple developers to work in parallel on branches, and enables merging changes together with history.

git merge: Combines two branches by creating a new "merge commit" that ties both histories together. Preserves complete branch history. Safer and transparent.

git rebase: Moves or replays your commits on top of another branch, creating a linear history. No merge commit. Creates a cleaner history but rewrites commit hashes — dangerous on shared/public branches.

Rule: Merge for shared branches (main, develop). Rebase for cleaning up local feature branches before merging.

Q24 Walk me through your TrustLens project architecture. Critical
